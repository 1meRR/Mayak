# Архитектура MVP

## 1. Общая схема

```text
Flutter Client A  ----\
                        \
                         > Rust Signaling Server <---- Flutter Client C
                        /
Flutter Client B  ----/
```

После обмена signaling-сообщениями клиенты создают прямые WebRTC peer-to-peer соединения.

## 2. Что децентрализовано здесь

Децентрализованная часть:
- медиа-потоки аудио/видео
- peer-to-peer transport между участниками
- нет медиасервера в середине

Не децентрализовано пока:
- discovery / signaling
- хранение недавних комнат
- текстовый чат комнаты
- identity management

## 3. Почему выбран WebRTC mesh

Плюсы:
- быстрый MVP
- нативная поддержка аудио/видео
- хорошая кроссплатформенность

Минусы:
- плохо масштабируется на большие комнаты
- каждый клиент отправляет поток каждому участнику

Практический предел для MVP:
- 1:1 отлично
- 3–4 участника приемлемо
- 5+ уже зависит от устройств и сети

## 4. Сообщения signaling-протокола

Основные типы:
- `join`
- `existing_peers`
- `peer_joined`
- `offer`
- `answer`
- `ice_candidate`
- `chat`
- `leave`
- `peer_left`

## 5. Почему Rust здесь на сервере

- предсказуемая производительность
- хороший async стек
- безопасная работа с конкурентным доступом к комнатам
- можно дальше эволюционировать в единое protocol/core-ядро

## 6. Что делать дальше для настоящей decentralization

### Шаг 1
Оставить signaling-сервер, но вынести:
- identity
- криптографию
- message log
- device sync

в общее Rust-ядро.

### Шаг 2
Добавить:
- libp2p discovery
- DHT/bootstrap nodes
- signed peer identities
- репликацию message state

### Шаг 3
Сделать гибрид:
- P2P для сообщений и 1:1
- SFU для больших групповых видеозвонков

Это обычно наиболее жизнеспособная архитектура.
